# levilamina-rs — Design Notes

This document records the load-bearing design decisions and the rules for
evolving the ABI. Source of truth for the ABI itself: `src/LeviRsAbi.h`,
mirrored field-for-field by `crates/levilamina-sys`.

## 1. Why a loader mod + custom mod type

LeviLamina's `ModManagerRegistry::addManager()` is public LLAPI, and
`ModRegistrar::loadAllMods()` resolves the manager for each manifest **at
dispatch time** (`ModManagerRegistry::loadMod` checks `hasManager(type)` when
the mod's turn comes), after topologically sorting mods by their
`dependencies`. Therefore:

1. `levilamina-rust-loader` is a normal native mod; its `ll_mod_load` registers
   a `RustModManager` for type `"rust"`.
2. Every Rust mod declares `"dependencies": [{"name": "levilamina-rust-loader"}]`,
   which guarantees the loader runs first in the same load pass.

This makes Rust mods first-class: dependency sorting, enable/disable, unload,
and mod listing all work through the standard machinery. This is the same
mechanism script-engine mods use.

## 2. Function-table injection, not import libraries

The loader resolves exactly one symbol in the Rust cdylib —
`levi_rs_main(const LeviRsApi*, LeviRsModHandle, LeviRsModVTable*)` — and hands
over a table of function pointers. Rationale:

- No import-library / link-time coupling between rustc (MSVC ABI) and
  clang-cl output; works identically on Linux later.
- The table pointer is versioned (`abi_version`, `struct_size`), so
  loader/mod mismatches fail fast with a clear error instead of UB.
- Rust mods never call C++ symbols directly and never own C++ heap memory, so
  LeviLamina's unified-allocator contract (`ll_memory_operator_overrided`) is
  satisfied by the loader alone — Rust uses its own allocator on its side of
  the fence.

## 3. String & memory conventions

- All strings are UTF-8 `(ptr, len)` views (`LeviRsStr`), no NUL requirement.
- Strings passed into a callback are valid only for the duration of the call.
- Data flowing Rust → C++ uses **sink callbacks invoked within the call frame**
  (`LeviRsStrSink`, `LeviRsCmdOutputSink`). No allocation ever crosses the
  boundary in either direction; there is no `free` function in the ABI by
  design.

## 4. Threading contract

- All callbacks (lifecycle, events, commands, scheduled tasks) are invoked on
  the **server thread**.
- `log`, `gaming_status`, `schedule`, `schedule_after` are thread-safe; the
  latter two are implemented on `ll::thread::ServerThreadExecutor` and are the
  designated way for background threads (Tokio runtimes, agents, I/O) to
  re-enter the game. Everything else is server-thread-only.
- The safe crate encodes this: `Server::subscribe_event` etc. are expected to
  be called from hooks (already on the server thread); `Server::get()` +
  `schedule` is the documented cross-thread pattern.

## 5. The universal event channel

`ll::event::DynamicListener` wraps `std::function<void(CompoundTag&)>`: the bus
serializes the concrete event into a `CompoundTag`, and deserializes the
(possibly modified) tag back into the event after the callback returns.
`Cancellable<E>::serialize` includes the cancelled flag (`serializeWithCancel`),
so cancellation is just a field write.

The bridge converts `CompoundTag ⇆ SNBT` (`Tag::toSnbt(SnbtFormat::Minimize)` /
`CompoundTag::fromSnbt`) and exposes subscribe-by-id with two conveniences:

- exact id match first, then **unique-suffix** fallback (`"PlayerChatEvent"`
  resolves if unambiguous) — event ids are derived from C++ type names
  (`ll::event::…`, possibly including inline namespaces, which vary), so
  suffix matching plus the in-game `/levirs events` dump avoids a whole class
  of user friction;
- listeners are owned per-mod and force-detached on unload, before the dylib
  is freed.

Trade-off, accepted for v0.1: SNBT round-trip costs a serialization per
listener call. Correctness first; a structured/zero-copy view is roadmap work
and can be added as new ABI entries without breaking v1.

## 6. Commands

- `register_command` uses LeviLamina's runtime command API
  (`getOrCreateCommand` + `runtimeOverload().optional("args", ParamKind::RawText)`),
  i.e. every Rust command is `/name [raw text]` in v0.1. Typed parameters are
  roadmap (they map cleanly onto more runtime overloads).
- Bedrock cannot unregister commands, so command **bindings** are stored in a
  loader-side table keyed by name; when a mod disables or unloads, its bindings
  are muted/nulled and the command answers with an error. Re-enabling or
  reloading the mod rebinds the same command.
- `execute_command` runs through `CommandRegistrar::executeCommand` with a
  console-level `ServerCommandOrigin` and returns collected output + success.

## 7. Panic and error policy

- Rust side: every `extern "C"` entry generated by the `levilamina` crate is
  wrapped in `catch_unwind`; panics are logged through the mod's logger and
  converted to `false`/no-op. A Rust mod must never unwind into C++.
- C++ side: `register_command` guards LeviLamina calls with try/catch and
  reports through the mod logger; load/unload paths return `ll::Expected<>`
  errors that surface in the standard mod-loading diagnostics.

## 8. ABI evolution rules (v1 → v2 …)

1. `LeviRsApi` and `LeviRsModVTable` are **append-only**: new function pointers
   go at the end; existing fields are never reordered, removed, or retyped.
2. Additive changes (new trailing fields) do **not** bump `abi_version`;
   consumers must check `struct_size` before using trailing fields.
3. Version acceptance is a **range, not exact equality**. The loader accepts a
   mod whose `abi_version` is in `[LEVI_RS_ABI_MIN_SUPPORTED, LEVI_RS_ABI_VERSION]`
   (a newer loader runs an older mod, since the table only grows additively and
   the old mod calls a byte-identical prefix); the mod accepts any loader whose
   `abi_version >= ` its own, with `struct_size` as the precise forward-compat
   gate. A breaking change (semantics, signature, field reorder/removal) bumps
   `LEVI_RS_ABI_VERSION` **and** raises `LEVI_RS_ABI_MIN_SUPPORTED` to that
   version, so tables that are no longer a prefix are refused.
4. The C header and the Rust `-sys` crate change in the same commit, always.

## 9. Deliberately out of scope for v0.1

- Direct pointer access to game objects (Actor*, Player*, BlockSource*):
  needs a handle-lifetime story; planned behind opaque generation-checked ids.
- Hooking arbitrary BDS functions from Rust: possible in principle via
  preloader's C symbol resolution, but it bypasses LeviLamina's abstractions;
  not this project's job.
- Client platform.
